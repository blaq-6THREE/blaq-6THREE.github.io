What then is a READme file?

A READme file is a text file (.txt, .md) that contains information about the software created for you (future you), the client, user, and anyone who could be a beneficiary of the software at hand. All for the user.

That is the importance of the documentation — the purpose of READMEs. Documenting your code, beit commenting of writing READmes is just like a clean cookbook. When thourghly written, it provides for the user or the implementator a stronger foundation to better feel as to how well to use the software or program at hand. I exists to help humans make sense of the code that we have written.

To be specific, READme as a documentation is to humans as code is to computers — well, your future self, a future employer, and many others. However, when writing a READme file, you should write them as though a 5 year old was going to use them as a manual before using your software.

You will want it to be easy for anyone to dive into your code and get it up and running without any hiccups!

Anatomy of a READme file

Title:

Well, this is what the name says it is.

Description:

With this a sentence or two would be good. Be sure to put enough context to capture the spirit of your project to get the user up and running.

Installation:

Provide for the user the “HOW" to install your program. This step is most critical as it is what will lead the way to how to better use your program.

Here, you include step by step process to install the application.

Make mention of any needed dependancies and/or libraries that should be installed before installation.

Also, be sure to specify the process if they are Operating System specific. That is, consider the main OS that your program would run on and give related details for them.

BEWARE: Users may have a hard time with this part of of your READme. Be sure to use simple vocabulary and DO NOT FORGET. Consider every step in this part.

Useage:

Once you’re confident about the steps provided in the installation process you provide them with using the application.

Here you provide examples must be included to show the proper use of your application.

Bugs:

Here you do what most us us artists, aka CS majors hate to do. You must include the possible bugs in your code.

Only you can provide this since you wrote your code and what state it currently is in.

License

Sooner or later your application, be it school project or personal prjects may grow and lisencing will be needed.

By default you have every right to your code, but including a license is important. This is becase of future contributions. Your project may be open to contributions in which case it will be super important to include a license.
